CORE Phases

Every issue passes through 4 mandatory cognitive phases. An AI Judge validates quality between each phase — if rejected, the phase is re-executed (up to 3 retries).

Phase What the Agent Does What the AI Judge Evaluates
Scan Reads the issue, maps the codebase — files, modules, patterns, dependencies Enough context? All relevant files identified?
Investigate Searches docs, APIs, best practices; analyzes complexity and duplication Research thorough? Reliable sources? Duplication checked?
Draft Plan Defines architecture, lists files to create/modify, builds TODO and test plan Plan covers all requirements? Architecture coherent?
Build Generates production code + automated tests in batches Code follows plan? All deliverables covered?

Quality Assurance

All generated code runs through a validation pipeline inside an isolated Docker container before any commit.

Stage What It Does
Commit Sanitizer Fixes invalid imports, smart-merges shared files (router.py, config.py, requirements.txt), protects repo files from overwrite
Syntax Check Parses all Python files (AST) to catch syntax errors
Import Check Verifies all imports resolve to existing modules
Load Test Runs from app.main import app — verifies FastAPI starts
Pytest Executes generated unit tests with coverage
SAST Bandit (vulnerabilities) + Ruff (lint/formatting)

Known Limitations & Roadmap

This is an advanced POC. These are the known limitations being actively worked on.

Test file truncation High

The LLM has an output token limit. Long test files (8+ tests) get cut mid-generation, causing syntax errors. Mitigation: prompt instructs max 5 tests per file, but the LLM may ignore this.

Tests with wrong repo imports High

The LLM generates tests that import repo modules with incorrect paths (e.g., from app.database import get_db when the real path is different). Auto-fix now receives dynamic context from repo files, but initial generation still depends on the LLM getting it right.

Auto-fix recreates repo files Medium

When tests fail, auto-fix may recreate existing repo files (main.py, router.py, database.py) with incomplete versions, causing error loops. The sanitizer protects shared files, but auto-fix sometimes generates versions that conflict with the merge.

Shared files loop in auto-fix Medium

The router.py is a shared file protected by the sanitizer. Auto-fix generates a version with correct imports, but the sanitizer merges it with the repo version producing broken code (e.g., articles.router without importing articles). Result: 5-attempt loop without converging.

Tests generated inside build Low

SKIP_TESTS=true skips the separate test generation call, but the build LLM may include test files in its output. These tests aren't filtered and can cause QA failures.

Code Review criticals don't become failures Low

When the Code Review LLM reports "critical" severity issues, they aren't converted into failures for auto-fix. Result: the repair node receives an empty failure_log and fixes nothing, entering a loop.
